## 模拟器实现说明

模拟器流程：

1. 读取 JSONL trace，并按 `arrival_order` 排序。
2. 将 prompt 用稳定正则 tokenizer 转成 token id。
3. 调用 `RadixCache.match_prefix` 统计命中 token。
4. 调用 `RadixCache.insert` 插入完整 prompt。
5. 如果 `cache.total_size() > pool_size`，调用 `cache.evict` 释放溢出 token。
6. 汇总 `prompt_tokens`、`cached_tokens`、`hit_rate`、`evict_calls` 等指标。

当前 tokenizer 是本地稳定 tokenizer，不是 Qwen tokenizer。它适合 A4 机制验证，但 GPU 前需要注意：真实 tokenizer 会改变绝对 token 数，可能需要重跑模拟矩阵。

## 与 A3 预测的对照

### W1

A3 预测：共享前缀多，策略差异较小。

A4 结果：在 45k 以上池大小，四个策略完全持平；在极小 10k 池下，LFU/SLRU 明显更好。

解释：W1 的共享前缀确实被结构保护，所以正常池大小下差异消失。10k 是极端小池，频率信号开始影响可淘汰叶子选择。

### W2

A3 预测：LRU/SLRU 应优于 FIFO，LFU 不一定稳定占优。

A4 结果：LFU/SLRU 明显优于 LRU/FIFO。

解释：当前合成 W2 的多轮历史会让同一会话前缀反复插入，`hit_count` 信号被强化。因此这个 W2 负载不只是“近期性主导”，也包含很强的频率重复。后续如果要更纯粹测试 LRU，需要调整 W2 的会话调度，让近期性更强、频率差异更弱。

### W3

A3 预测：LFU/SLRU/2Q 应优于 FIFO，可能优于 LRU。

A4 结果：小池和中池下 LFU/SLRU 优于 LRU/FIFO；池足够大时全部持平。

解释：符合 Zipf 热点假设。池足够大时不再强淘汰，策略差异自然缩小。

### W4

A3 预测：2Q 应显著优于 LRU，LRU 可能被扫描流量污染。

A4 结果：当前还没有 2Q，内置策略中 LFU/SLRU 优于 LRU/FIFO。

解释：扫描流量确实给 LRU/FIFO 压力，但 LFU/SLRU 的频率保护能保住部分热点。2Q 是否进一步优于 SLRU，要等 A5 实现后再补矩阵。

## 关键发现

1. A4 模拟器已经能跑通完整 main trace。
2. W1 验证了“共享前缀使策略差异变小”的判断。
3. W3/W4 验证了频率和保护段策略在 Zipf/扫描场景下的优势。
4. W2 当前设计偏向频率重复，不是纯近期性负载；这是 A4 帮我们提前发现的问题。
5. 2Q 还未接入，因此 A4 当前只是内置策略基线矩阵。

## 后续修正建议

进入 A5 前，可以保留当前 W2，因为它能稳定制造策略差异。

但如果最终报告想突出“多轮对话近期性”，建议后续增加一个 W2b：

- 减少同一会话的 hit_count 放大；
- 使用更强的最近会话局部性；
- 让 LRU 和 SLRU 的优势更容易显现。

当前 A4 不修改 `notes/03-predictions.md`，预测偏差只在本总结中记录。

## 限制

1. 当前 tokenizer 是 `regex_hash`，不是 Qwen tokenizer。
2. 当前模拟是顺序 replay，没有并发和 `lock_ref` 干扰。
3. 当前矩阵没有 2Q，需要 A5 补齐。
4. 当前池大小是 token 数粗设，GPU 阶段要用 `max_total_num_tokens` 校准。
