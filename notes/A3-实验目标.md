# A3 实验目标：负载设计与 trace 生成

## 本阶段要解决的问题

A3 要把前面 A1/A2 的机制理解转化为可运行的请求序列，也就是 trace。

我们不是随便造请求，而是要设计四类访问模式，让每一类负载都对应一个明确的缓存机制假设：

- 共享前缀多时，策略差异是否会变小；
- 多轮对话这类深而窄的树中，近期性策略是否更占优；
- Zipf 多租户场景中，频率型策略是否更占优；
- 扫描污染场景中，2Q 是否能保护真正热点。

## 需要完成的任务

1. 设计 W1-W4 四类负载。
2. 为每类负载写可复现的 trace 生成逻辑。
3. 固定随机种子，保证每次生成结果一致。
4. 输出统一 JSONL 格式，供 A4 模拟器和 B 阶段 replay 脚本复用。
5. 写下实验前预测，形成 `notes/03-predictions.md`。

## Trace 统一格式

每条请求至少包含：

```json
{"request_id": "...", "prompt": "...", "max_tokens": 1, "arrival_order": 0}
```

生成器可以额外附带元数据，例如 workload、tenant_id、conversation_id、approx_prompt_tokens 等，方便后续分析。

## 本阶段产出

- `traces/generate_traces.py`：统一 trace 生成脚本。
- `configs/trace_plan.json`：A3 trace 规模与 seed 配置。
- `notes/03-predictions.md`：先验预测文档。
- `notes/A3-实验总结.md`：本阶段总结。

生成出来的 `traces/*.jsonl` 不入库，因为 trace 可能包含数据集内容或体积较大。仓库只提交生成脚本、固定 seed 和配置。

## 对整个实验的作用

A3 决定后续实验是否有解释力。

如果负载设计不好，即使 GPU 跑出数据，也很难回答“为什么某个策略好”。A3 的目标是让每个负载都有明确机制指向，这样 A4 的离线模拟和 B 阶段真机结果才能被解释。
