# A6 实验目标：客户端 replay、结果汇总与矩阵运行脚本

## 本阶段要解决的问题

A6 的目标是把后续 GPU 实验前的客户端评测链路准备好。也就是说，在真正租卡之前，我们需要有一套可以对 SGLang 服务发请求、逐条记录指标、汇总结果、批量运行矩阵的脚本。

这一阶段仍然优先离线完成：先用本地 mock HTTP 服务模拟 SGLang `/generate` 返回结构，验证 replay、并发、异常处理和结果解析都能工作。等进入 B 阶段时，只需要把 base URL 换成真实 SGLang server。

## 需要完成的任务

1. 实现 `bench/replay.py`：按 trace 顺序向服务端发送请求，支持可控并发，记录逐请求 JSONL。
2. 实现 `bench/mock_server.py`：提供一个轻量 mock `/generate` 和 `/flush_cache` 服务，用于无 GPU 调试。
3. 实现 `bench/collect.py`：汇总 raw JSONL，输出命中率、TTFT 分位数、总时延分位数、吞吐等指标。
4. 实现 `scripts/run_matrix.sh`：参数化运行策略、负载、压力或池大小组合，每组前调用 `/flush_cache`。
5. 增加必要的配置文件，让 A6 脚本能复用 A3/A4 的 trace 与策略矩阵。
6. 用 smoke trace + mock server 完成本地验证。

## 关键指标

逐请求记录至少包含：

- `request_id`
- `workload`
- `policy`
- `prompt_tokens`
- `cached_tokens`
- `hit_rate`
- `ttft_ms`
- `latency_ms`
- `ok`
- 错误信息或 HTTP 状态码

汇总结果至少包含：

- 总请求数、成功数、失败数
- `sum(cached_tokens) / sum(prompt_tokens)` 命中率
- TTFT 的 p50、p90、p99
- 总时延的 p50、p90、p99
- 总 token 吞吐与请求吞吐

## 对整个实验的作用

A6 是从 CPU 模拟阶段走向 GPU 真机实验的桥。

A1-A5 已经解决“应该测什么、怎么预测、2Q 怎么实现”的问题；A6 要解决“租卡后怎么稳定、可复现、低浪费地测”的问题。只要 A6 在 mock 服务上跑通，B 阶段就不需要临场写评测脚本，可以把 GPU 时间集中花在验证缓存策略本身。
