# 简历项目描述

## 项目名称

SGLang KV Cache 淘汰策略实验与 2Q 策略实现

## 简历版本

基于 SGLang 0.5.14 完成 KV Cache 淘汰策略优化实验，阅读并改造 RadixCache 与 eviction policy 源码，在 `evict_policy.py` 中实现 `TwoQStrategy`，并在 `radix_cache.py` 的 insert、hit、split、evict 生命周期中接入策略 hook；扩展策略注册与 CLI 参数，使真实服务支持 `--radix-eviction-policy 2q` 启动。构建 W1-W4 多类请求 trace，覆盖 few-shot 前缀复用、多轮对话、Zipf 热点租户与 hot -> scan -> hot 扫描污染场景；开发 replay、summary、scan-revisit 分析脚本，在 RTX 3090 + Qwen2.5-7B-Instruct 上采集 cached_tokens、hit rate、TTFT 等指标。实验发现当前 2Q 在扫描污染压力下略优于 LRU、与 SLRU 基本持平，而 LFU 在稳定高频热点场景下表现最佳，并据此分析出当前 2Q 实现缺少严格 A1in/Am 容量配额、行为接近 SLRU-like 的原因。

## 可拆分要点

- 深入分析 SGLang RadixCache、KV cache eviction policy 与 CLI 参数注册流程，实现 `TwoQStrategy` 并完成真实 GPU 服务接入。
- 在 RadixCache 插入、命中、拆分、淘汰路径中接入 `on_insert`、`on_hit`、`on_split`、`on_evict` hook，使策略状态跟随前缀树节点变化。
- 构建可复现实验框架，支持 trace 生成、SGLang `/generate` 回放、cache flush、raw JSONL 记录与 CSV 汇总。
- 设计 W4 pressure workload，通过 hot -> scan -> hot 回访模式评估扫描污染下的热点保护能力。
- 编写 scan-revisit 分析脚本，从 raw JSONL 中拆出 scan 后 32 条热点请求，比较 cached_tokens、hit_rate、TTFT，避免全局平均值掩盖策略差异。
- 在 Qwen2.5-7B-Instruct 上完成 LRU、SLRU、LFU、2Q 对比实验，总结不同策略在稳定热点与扫描污染场景下的适用边界。

## 面试展开点

1. 为什么 KV cache 命中会影响 TTFT。
2. SGLang RadixCache 如何利用公共前缀复用 KV。
3. LRU、SLRU、LFU、2Q 的淘汰信号分别是什么。
4. `TwoQStrategy.get_priority`、`on_insert`、`on_hit`、`on_evict` 分别做了什么。
5. 为什么当前 2Q 和 SLRU 的实验结果接近。
6. 为什么 LFU 在稳定高频热点场景下反而更强。
7. 如果继续优化，如何实现严格 A1in/A1out/Am 配额和 ghost hit 统计。
