# A2 实验总结：KV 显存账本与模型选型

## 本阶段完成的工作

本阶段完成了 KV cache 显存账本，并确定了后续实验的模型选择。

具体完成事项：

1. 明确了每 token KV cache 的计算公式。
2. 核对了三个候选模型的结构参数。
3. 计算了每个候选模型的每 token KV 占用。
4. 粗估了 24GB GPU、`mem_fraction_static=0.85` 下的 KV 池大小。
5. 给出了主实验模型和调试模型选择。

## 关键计算结果

| 模型 | 每 token KV | 粗估可缓存 token | 用途判断 |
|---|---:|---:|---|
| Llama-2 7B | 512 KiB | 约 1.3-1.4 万 | KV 太大，否决主实验 |
| Qwen2.5-7B-Instruct | 56 KiB | 约 9 万 | 主实验模型 |
| Qwen2.5-1.5B | 28 KiB | 约 60 万 | 调试和快速扫参 |

## 关键结论

主实验选择 **Qwen2.5-7B-Instruct**。

原因：

- 它是 7B 级模型，实验说服力比 1.5B 更强；
- 它使用 GQA，KV cache 比 Llama-2 7B 的 MHA 结构小很多；
- 24GB 卡上约 9 万 token 级别的 KV 池，适合构造 2-5 倍工作集压力。

调试模型选择 **Qwen2.5-1.5B**。

原因：

- 权重小，启动快；
- KV 池大，适合先验证服务、脚本和并发逻辑；
- 可以低成本跑小规模矩阵，提前发现脚本问题。

不建议使用 **Llama-2 7B** 做主实验。

原因：

- 每 token KV 约 512 KiB；
- 24GB 卡上粗估只能缓存约 1.4 万 token；
- 容易导致缓存压力过于极端，策略比较失真。

## 对后续阶段的影响

A3 设计 trace 时，要围绕 Qwen2.5-7B-Instruct 的池大小设计工作集。

租卡前可先按 9 万 token 粗估：

- 中压：约 18 万 token 工作集；
- 高压：约 36-45 万 token 工作集。

GPU 冒烟阶段拿到真实 `max_total_num_tokens` 后，必须重新校准这些压力档。

## 风险与注意事项

1. A2 中的权重占用是粗估，不能当最终实验数据。
2. `mem_fraction_static` 圈定的是权重和 KV 池的总和，激活、运行时碎片和 backend 行为可能影响真实容量。
3. 最终容量必须以 SGLang 启动日志里的 `max_total_num_tokens` 为准。
4. 如果 GPU 阶段真实池大小和 A2 粗估偏差较大，要回到 A4 重新跑模拟器预测矩阵。
