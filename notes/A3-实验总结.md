## Trace 生成命令

生成 smoke trace：

```bash
python traces/generate_traces.py --profile smoke --workload all --output-dir traces
```

生成 main trace：

```bash
python traces/generate_traces.py --profile main --workload all --output-dir traces
```

## 四类负载设计

### W1 few-shot 评测

目的：模拟共享 5-shot 前缀的评测请求。

机制假设：共享前缀会成为内部节点，策略差异应该较小。

### W2 多轮对话

目的：模拟多个会话交错增长的上下文。

机制假设：树更深更窄，近期性更重要，LRU/SLRU 应该优于 FIFO。

### W3 Zipf 多租户

目的：模拟 system prompt 按 Zipf 热度分布被访问。

机制假设：热门租户反复出现，LFU/SLRU/2Q 应该更容易保留热点。

### W4 扫描污染

目的：在 Zipf 多租户基础上插入一次性长文档。

机制假设：LRU 容易被扫描流量污染，完整 2Q 应该更有优势。

## 重要取舍

当前 A3 使用确定性合成 trace，而不是直接下载 MMLU 和 ShareGPT。

原因：

1. 合成 trace 更适合先固定机制假设，避免数据清洗阻塞 A4。
2. 每条请求的结构、长度和访问模式都可控，方便调试模拟器。
3. 后续如果需要更贴近真实数据，可以在同一格式下增加 MMLU/ShareGPT 数据源适配器。

这个取舍需要在 README 或最终报告里说明，避免把合成负载说成真实数据集实验。

## 后续注意事项

1. A4 模拟器要用真实 tokenizer 重新计算 token 数，不能依赖 A3 的粗略 token 估计。
2. A4 跑完后，需要根据模拟结果检查 A3 的负载强度是否合适。
3. 如果某个负载策略差异太小，优先调整池大小曲线，而不是直接修改预测文档。
4. 如果后续接入 MMLU/ShareGPT，应保持 JSONL 字段兼容。
