# A4 实验目标：离线 trace 模拟器

## 本阶段要解决的问题

A4 要在不租 GPU 的情况下，先用 CPU 模拟不同淘汰策略在 W1-W4 trace 上的缓存命中率。

这个阶段的核心问题是：

```text
如果 KV 池大小固定，不同策略在同一条 trace 上能命中多少 prompt token？
```

## 需要完成的任务

1. 编写 trace-driven 离线模拟器。
2. 读取 A3 生成的 JSONL trace。
3. 将 prompt 转成稳定 token id 序列。
4. 复用 vendor 版 `RadixCache` 执行 `match_prefix -> insert -> evict`。
5. 支持多种淘汰策略和多档池大小。
6. 输出每个 workload × policy × pool_size 的预测命中率。
7. 生成 `results/sim_matrix.csv`。

## 本阶段的输入

- `traces/generate_traces.py`
- `configs/trace_plan.json`
- 本地生成的 `traces/w*_smoke_seed20260708.jsonl`
- 本地生成的 `traces/w*_main_seed20260708.jsonl`

生成的 JSONL trace 不入库，但可以通过脚本和 seed 复现。

## 本阶段的输出

- `simulator/run_simulator.py`
- `configs/sim_config.json`
- `results/sim_matrix.csv`
- `notes/A4-实验总结.md`

## 对整个实验的作用

A4 是租 GPU 前最重要的离线验证环节。

它能提前告诉我们：

- 哪些策略在某类负载上可能更好；
- W1-W4 的负载设计是否真的制造了策略差异；
- A3 的预测是否需要在理论阶段修正；
- 后续 GPU 实验应该优先验证哪些组合。

如果模拟器里某个负载完全跑不出策略差异，就应该先调整 trace 或池大小，而不是直接租 GPU 盲跑。
