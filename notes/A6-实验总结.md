新增文件：

## 2Q 状态确认

在进入 A6 前，我复核了 A5 的 2Q 实现。

当前实现已经满足本实验所需的“标准 2Q”核心：

- 有 A1in：新进入的试用队列。
- 有 A1out：只存 key fingerprint 的 ghost queue，不占 KV。
- 有 Am：确认复用后的主缓存区。
- A1in 淘汰会写入 A1out。
- A1out 命中后重新插入会晋升 Am。
- A1in resident hit 也会晋升 Am，这一点是面向 KV cache 场景的合理增强。
- Am 命中会刷新 LRU 顺序。
- Radix tree split 后会继承 2Q 段状态。

它与论文版或数据库 buffer manager 里的完整工程实现仍有差距：

1. A1in 没有单独 token 级配额，依赖全局 KV 池压力与优先级排序。
2. A1out 使用完整 token 路径作为 fingerprint，后续可替换为 page hash 或 RadixCache 已有 hash。
3. 暂未实现自适应调参或 ARC 式动态平衡。
4. 并发、`lock_ref` 和真实 GPU server 行为还需要 B 阶段校准。

这些差距不阻塞 A6，因为 A6 的任务是准备评测链路，而不是继续调优 2Q。

## replay.py

`bench/replay.py` 的输入是 A3 生成的 trace JSONL。它会向 SGLang native `/generate` 发送如下结构：

```json
{
  "text": "...",
  "sampling_params": {
    "max_new_tokens": 1,
    "temperature": 0.0
  },
  "model": "..."
}
```

逐请求输出包含：

- `request_id`
- `arrival_order`
- `workload`
- `policy`
- `pressure`
- `model`
- `prompt_tokens`
- `cached_tokens`
- `hit_rate`
- `completion_tokens`
- `ttft_ms`
- `latency_ms`
- `ok`
- `error`
- `status_code`

实现特意改为纯标准库，不依赖 `httpx`，这样在干净 WSL/GPU 机器上只要有 Python 就能先跑通基础链路。

## mock_server.py

`bench/mock_server.py` 提供三个端点：

- `GET /health`
- `POST /flush_cache`
- `POST /generate`

mock server 会用简单 tokenizer 计算 prompt token，并在已有 prompt 中找最长公共前缀作为 `cached_tokens`，返回形如：

```json
{
  "text": " mock",
  "meta_info": {
    "prompt_tokens": 247,
    "cached_tokens": 16,
    "completion_tokens": 1,
    "ttft_ms": 3.0
  }
}
```

它不是替代模拟器，只用于调通客户端解析、并发和汇总。

## collect.py

`bench/collect.py` 会按 `(model, policy, pressure, workload, trace)` 分组汇总 raw JSONL。

输出字段包括：

- 请求数、成功数、失败数
- prompt tokens、cached tokens、completion tokens
- `hit_rate = sum(cached_tokens) / sum(prompt_tokens)`
- TTFT p50/p90/p99
- latency p50/p90/p99
- wall time、requests/sec、tokens/sec

## run_matrix.sh

`scripts/run_matrix.sh` 使用环境变量控制矩阵：

- `BASE_URL`
- `MODEL`
- `PROFILE`
- `POLICIES`
- `WORKLOADS`
- `PRESSURES`
- `CONCURRENCY`
- `LIMIT`
- `OUTPUT_DIR`

示例：

```bash
BASE_URL=http://127.0.0.1:30000 \
PROFILE=main \
POLICIES="lru lfu fifo slru 2q" \
WORKLOADS="w1 w2 w3 w4" \
PRESSURES="medium high" \
CONCURRENCY=8 \
bash scripts/run_matrix.sh
```

注意：SGLang 的 eviction policy 通常是 server 启动参数，所以脚本会提示“当前 policy block 需要对应策略的 server”。也就是说，后续 B 阶段可以选择：

1. 每个策略手动或由外层脚本重启一次 server；
2. 同一策略 server 内跑完不同 workload/pressure，并在每组前 `/flush_cache`。
