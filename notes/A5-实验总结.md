## 2Q 策略实现

本实现使用三个区域：

- A1in：新插入节点先进入试用队列，淘汰优先级最高。
- A1out：从 A1in 淘汰的节点只留下完整 token 路径 fingerprint，不保存 KV。
- Am：被确认复用的节点进入主缓存区，淘汰优先级最低。

节点晋升规则：

- 新节点插入时，如果 fingerprint 不在 A1out，进入 A1in。
- 新节点插入时，如果 fingerprint 命中 A1out，进入 Am。
- A1in 节点在缓存内再次命中时，也晋升到 Am。
- Am 节点再次命中时刷新 LRU 时间。

这样做的原因是：在 KV cache 场景中，`match_prefix` 命中本身就是复用信号。如果只允许“淘汰后再插入”才能晋升，会低估 resident reuse，导致 2Q 在小池下过度接近 LRU。

## 接口扩展

为了表达有状态策略，本阶段给 `EvictionStrategy` 增加了四个向后兼容的可选钩子：

```python
on_insert(node)
on_hit(node)
on_evict(node)
on_split(parent_node, child_node)
```

已有 LRU/LFU/FIFO/SLRU 不实现这些钩子也不会改变行为。2Q 通过这些钩子维护节点所处队列、ghost history 和 split 后的状态继承。

## 性能修正

最初版本在 `get_priority(node)` 中通过遍历 `OrderedDict` 查找节点位置，导致 W2 大池模拟明显变慢。

最终版本改为维护单调递增的 `queue_clock`，节点进入 A1in/Am 或 Am 被命中刷新时写入 `node_order[node.id]`。因此 `get_priority(node)` 可以 O(1) 返回：

```python
(segment_rank, queue_time, node.last_access_time)
```

这让完整 main 矩阵可以正常跑完。

## 2Q 在 main 矩阵中的表现

2Q 命中率如下：

| workload | 10000 | 45000 | 90000 | 180000 |
| --- | ---: | ---: | ---: | ---: |
| w1 | 0.5258 | 0.8263 | 0.8263 | 0.8263 |
| w2 | 0.0354 | 0.0898 | 0.2056 | 0.4238 |
| w3 | 0.6564 | 0.8531 | 0.8595 | 0.8595 |
| w4 | 0.4859 | 0.6104 | 0.6341 | 0.6409 |

主要观察：

1. W1 小池下，2Q 明显优于 LRU/FIFO，但略低于 LFU/SLRU；池子达到 45000 后，各策略都达到同一上限。
2. W2 长上下文循环中，2Q 在 45000 是最佳策略，90000 和 180000 略低于 LFU/SLRU，但远高于 LRU/FIFO。
3. W3 混合前缀场景中，2Q 接近 LFU/SLRU，90000 以上达到同一上限。
4. W4 多租户混合场景中，2Q 始终明显优于 LRU/FIFO，接近 LFU/SLRU。

## 阶段结论

A5 证明了两件事：

1. 单纯的无状态 `get_priority(node)` 不足以表达完整 2Q，因为 2Q 需要记录 A1in、A1out、Am 之间的历史迁移。
2. 通过向 RadixCache 增加事件钩子，可以在不破坏已有策略的情况下实现有状态淘汰策略。

当前 2Q 已经具备可实验性：能运行 CPU 单测，能接入 trace simulator，能和 A4 的基线策略放在同一矩阵中对比。

## 当前限制

1. A1in 没有单独 token 配额，主要依赖全局 KV 池压力和优先级排序。
2. A1out 使用完整 token 路径 fingerprint，适合 CPU 模拟；后续上游化时可以考虑复用 page hash。
3. 本阶段仍是 CPU trace simulator 结果，还没有进入 GPU server 实测。
4. 2Q 参数尚未系统调优，后续可以围绕 A1out 大小、resident hit 晋升规则、split 状态继承做 ablation。
