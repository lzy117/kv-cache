# 实验思路总览与核心名词解释

这份笔记的目标是先把整个实验“为什么做、做什么、怎么做、结果怎么看”理顺。它不追求源码细节，更多是给后续 A1-A7 阶段建立直觉。

## 一句话概括这个实验

我们要研究：在 SGLang 的前缀缓存系统中，不同 KV cache 淘汰策略在不同请求负载下，谁能保留更多有用缓存、提高命中率，并解释原因。

更具体地说：

- SGLang 会把已经算过的 prompt 前缀缓存起来。
- 后续请求如果有相同前缀，就可以复用缓存，少算一部分 token。
- GPU 显存有限，缓存放满后必须淘汰一部分旧内容。
- 不同淘汰策略，例如 LRU、LFU、FIFO、SLRU、2Q，会决定“丢谁、留谁”。
- 我们要系统评测这些策略在不同访问模式下的效果。

## 为什么这个问题重要

大模型推理时，prompt 越长，prefill 阶段越贵。很多真实场景会重复使用相同或相似的前缀：

- few-shot 评测共享示例 prompt；
- 多轮对话反复带上历史上下文；
- 多租户应用反复使用同一个 system prompt；
- RAG 或长文档场景会插入大量一次性长文本。

如果前缀缓存命中，就能少做重复计算，降低 TTFT，也就是首 token 延迟。

但缓存不是无限的。显存放满后，系统必须决定淘汰哪些缓存。这个选择会直接影响后续请求还能不能命中。

## 什么是 KV cache

Transformer 推理时，每一层 attention 都会为 token 计算 Key 和 Value，简称 K/V。

对于已经处理过的 token，模型后续生成时不需要重新计算它们的 K/V，可以把它们存起来复用。这些存起来的 K/V 就叫 KV cache。

直觉上可以这样理解：

- prompt token 第一次进入模型时，会产生一批中间结果；
- 这些中间结果存在显存里；
- 后面如果还要基于同样的上下文继续算，就直接复用这批结果；
- 这能避免重复 prefill。

KV cache 通常很占显存。模型越大、层数越多、KV head 越多、上下文越长，占用越大。

## 什么是 KV 池

KV 池就是专门用来存 KV cache 的显存区域。

可以把它想成一个固定容量的仓库：

- 每存一个 token 的 KV，就占用仓库里一小块空间；
- prompt 越长，占用越多；
- 仓库满了，就必须搬走一些旧缓存；
- 淘汰策略决定先搬走谁。

在 SGLang 里，启动参数 `--mem-fraction-static` 会限制“模型权重 + KV 池”总共能使用多少 GPU 显存。所以 KV 池大小大致可以估算为：

```text
KV 池大小 ≈ GPU 显存 × mem_fraction_static - 模型权重占用
```

例如 24GB 显卡，`mem_fraction_static=0.85` 时，权重和 KV 池合计大约最多用 20.4GB。模型权重越大，留给 KV 池的空间越小。

这就是为什么 A2 要做显存账本：我们必须先知道仓库大概有多大，才能设计合适的实验压力。

## 什么是前缀缓存

前缀缓存是 KV cache 的一种使用方式。

如果两个请求有共同 prompt 前缀：

```text
请求 1: 你是一个数学老师。请解答：1+1=?
请求 2: 你是一个数学老师。请解答：2+2=?
```

它们共享前缀：

```text
你是一个数学老师。请解答：
```

这个共享前缀的 KV cache 可以复用。请求 2 不必从头重新计算整个 prompt，只需要复用前缀部分，再计算后面不同的部分。

SGLang 的 RadixAttention / RadixCache 就是为了高效管理这种共享前缀。

## 什么是 RadixCache

RadixCache 可以理解为一棵前缀树。

它把 token 序列按公共前缀组织起来：

```text
root
 └── 公共 system prompt
      ├── 用户 A 的问题
      └── 用户 B 的问题
```

这样做的好处是：

- 相同前缀只存一份；
- 查询时可以快速找到最长可复用前缀；
- 不同请求的分叉部分挂在不同叶子上。

这个实验里非常关键的一点是：SGLang 的 RadixCache 主要只能淘汰叶子节点。内部节点如果还有子节点，通常不能直接淘汰。

这会影响实验结论：很多热门公共前缀会因为变成内部节点而被结构性保护，不完全由 LRU/LFU 这些策略决定。

## 什么是缓存命中

缓存命中指：一个新请求来了以后，它的 prompt 有多少 token 可以直接从已有 KV cache 中复用。

例如：

```text
prompt_tokens = 1000
cached_tokens = 700
```

说明这个请求有 700 个 token 命中了缓存，只需要重新计算剩下 300 个 token。

常用命中率可以写成：

```text
命中率 = cached_tokens / prompt_tokens
```

实验里我们会关心：

- 总体命中率；
- 不同策略之间的命中率差异；
- 命中率变化是否能解释 TTFT 改善；
- 离线模拟器预测的命中率和真机实测差多少。

## 什么是淘汰策略

KV 池满了以后，系统要删除一部分旧缓存来腾空间。删除谁，就是淘汰策略的问题。

常见策略：

- LRU：Least Recently Used，淘汰最久没访问的缓存。
- LFU：Least Frequently Used，淘汰访问次数最少的缓存。
- FIFO：First In First Out，先来的先淘汰。
- SLRU：把缓存分成试用段和保护段，访问次数够多的进入保护段。
- 2Q：用 A1in、A1out、Am 三个区域区分“一次性扫描流量”和“真正热点”。

这个实验的核心不是简单说哪个策略最好，而是回答：

```text
什么负载下，什么策略更合适，为什么？
```

## 什么是 2Q

2Q 是一种带“试用”和“幽灵历史”的缓存淘汰策略。

可以粗略理解成：

- 新来的缓存先进入 A1in，也就是试用队列。
- 如果它很快被淘汰，只在 A1out 幽灵队列里留下记录，不继续占 KV 空间。
- 如果它之后又来了，说明它可能是真的热点，就晋升到 Am 主缓存区。
- Am 里的内容再按 LRU 管理。

2Q 的价值是抵抗扫描污染。

所谓扫描污染，就是大量只访问一次的长文本把缓存池洗掉。例如突然插入一批一次性长文档请求，它们可能占掉很多 KV 池。如果用普通 LRU，这些新来的长文档可能把原本有用的热点 system prompt 挤出去。2Q 希望让这些“一次性流量”先经过试用区，不要轻易污染主缓存区。

## 什么是 trace

trace 是一组固定下来的请求序列。

每条请求通常包含：

```json
{"request_id": "...", "prompt": "...", "max_tokens": 1, "arrival_order": 123}
```

我们用 trace 来保证实验可复现。所有策略都跑同一批请求，顺序也一样，这样结果才公平。

如果一边跑一边让模型随机生成新对话，再把生成结果拼回历史，不同策略看到的 prompt 内容就可能不同，实验就不可比。所以手册强调：多轮对话 trace 要用数据集里的原始回复拼接，不能实时生成。

## 什么是负载

负载就是请求的访问模式。

本实验设计四类负载：

| 负载 | 场景 | 主要特征 | 预期 |
|---|---|---|---|
| W1 | few-shot 评测 | 共享前缀多 | 策略差异小 |
| W2 | 多轮对话 | 深而窄，近期性强 | LRU/SLRU 可能好 |
| W3 | 多租户 system prompt | 热门租户反复出现 | LFU/2Q 可能好 |
| W4 | 扫描污染 | 一次性长文档冲击缓存 | 2Q 应该更有优势 |

这些负载不是随便造的，而是分别对应不同缓存机制假设。后面实验数据要验证这些假设。

## 什么是离线模拟器

离线模拟器是在 CPU 上模拟 RadixCache 行为的工具。

它不真的跑大模型，也不真的占 GPU 显存，而是：

1. 读取 trace；
2. 把 prompt 转成 token；
3. 用 RadixCache 查最长命中前缀；
4. 模拟插入和淘汰；
5. 统计不同策略的预测命中率。

离线模拟器的意义是：先在 CPU 上预测趋势，再决定 GPU 实验怎么跑。

这能避免租 GPU 后盲目试错。GPU 阶段应该只验证已经在 CPU 上想清楚、跑通过的实验。

## 整个实验按什么顺序推进

### 阶段 A：离线准备

目标：不用 GPU，把理论、代码、trace、模拟器、2Q 设计都准备好。

包括：

- A1：读懂 RadixCache 和淘汰策略机制；
- A2：算清 KV 显存账本，确定模型和 KV 池尺度；
- A3：设计四类负载，生成固定 trace；
- A4：做离线 trace 模拟器；
- A5：实现完整 2Q，并接入模拟器；
- A6：写 replay/collect 脚本；
- A7：租卡前 gate，确认所有离线工作完成。

### 阶段 B：GPU 实验

目标：用真机验证离线预测。

主要做：

- 启动 SGLang 服务；
- 记录真实 `max_total_num_tokens`，也就是实际 KV 池 token 容量；
- 跑策略 × 负载 × 压力矩阵；
- 收集 `cached_tokens`、`prompt_tokens`、TTFT、吞吐；
- 对照离线模拟器结果。

### 阶段 C：分析成稿

目标：把实验结果变成能讲清楚的项目成果。

主要输出：

- 主结果表；
- 命中率曲线；
- 预测 vs 实测误差；
- W4 中 LRU 被扫描洗掉、2Q 保住热点的机制图；
- README、简历表述和面试问答。

## A2 为什么是下一步

A2 要解决的是“KV 池到底有多大”。

如果不知道 KV 池大小，就无法设计合理 trace：

- trace 太小：缓存永远放得下，不发生淘汰，策略差异看不出来。
- trace 太大：缓存被极端冲爆，所有策略都表现很差，结论也失真。
- 模型选错：例如 MHA 模型的 KV 太大，会让 24GB 卡上的缓存池过小，实验不稳定。

A2 会帮我们确定：

- 主实验用哪个模型；
- 调试用哪个模型；
- 每 token KV 大概占多少显存；
- 24GB 卡上大约能缓存多少 token；
- 后续 W1-W4 trace 应该设计成 KV 池的几倍压力。

一句话：A2 是把实验从“概念设计”落到“可计算尺度”的步骤。
